* Purpose

Distributed systems orchestration framework in GNU Guile 3.0+, exploring
actor-model concurrency, event sourcing, and OTP-style supervision patterns
in Scheme. This is a scratch/experimental repository.

* Goals

- Implement core distributed systems primitives in idiomatic Guile Scheme
- Provide actor-based concurrency with supervision trees
- Support content-addressable storage with write-ahead logging
- Enable distributed coordination (leader election, locks, vector clocks)
- Target FreeBSD as primary platform (macOS for development)

* Non-Goals

- Production readiness (this is a scratch repo)
- Guile Fibers integration (uses raw threads currently)
- Network transport (coordination is in-process only)
- Real Raft/Paxos consensus (election is simplified/simulated)

** Concurrency Model

All concurrency uses ~(ice-9 threads)~ with mutex/condvar channels.
Guile Fibers are listed as a future goal but not integrated.
The actor system uses one OS thread per actor via ~call-with-new-thread~.

* Key Design Decisions

- *Channels over Fibers*: Uses hand-rolled channel type (mutex + condvar + list)
  rather than Fibers channels, keeping the dependency footprint minimal.
- *Content-addressable storage*: Values are SHA256-hashed (stub implementation)
  and stored by hash. WAL provides crash recovery.
- *OTP supervision*: Three restart strategies (one-for-one, one-for-all,
  rest-for-one) with restart-count limits and time windows.
- *Circuit breaker*: Errors module includes a circuit breaker with
  closed/half-open/open states and configurable threshold/timeout.
- *Literate programming*: Build system expects org-mode files tangled via Emacs,
  though source files also exist directly in src/.
- *core vs core-enhanced*: Two orchestrator implementations coexist. ~core.scm~
  is minimal; ~core-enhanced.scm~ adds config, metrics, lifecycle states,
  scheduler, dispatcher, and monitor threads. They share the same module
  name pattern but ~core-enhanced~ redefines channel primitives locally.

* Current Status

- *Phase*: Experimental / scratch
- *Completeness*: Core modules implemented; no network transport, no real
  distributed operation (coordination is in-process simulation)
- *CI*: GitHub Actions pipeline for Ubuntu + FreeBSD (tests run with ~|| true~)
- *Platform*: Targets FreeBSD 14+; Makefile hardcodes ~guile3~/~guild3~

* Known Limitations

- ~core.scm~ and ~core-enhanced.scm~ both define orchestrator types with
  conflicting exports (~make-orchestrator~, channel functions)
- SHA256 in storage.scm is a stub (sum-based, not cryptographic)
- No real network layer -- coordination module simulates votes with ~random~
- Priority queue in core-enhanced is a stub (empty list)
- Fibers not used despite README mentioning them as a key feature
- CI tests use ~|| true~ so failures are silent
- No ~guix.scm~, ~LICENSE~, or ~CONTRIBUTING.org~ despite README references
