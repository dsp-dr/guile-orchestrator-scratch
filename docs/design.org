* System Architecture

** Overview

The Guile Orchestrator follows an actor-based architecture pattern, leveraging Guile's Fibers library for lightweight concurrency. The system is designed to be:

- *Distributed*: Can coordinate tasks across multiple nodes
- *Fault-tolerant*: Handles actor failures gracefully
- *Scalable*: Actors can be dynamically spawned based on load
- *Functional*: Emphasizes immutability and pure functions

* Design Patterns

** Actor Model Implementation

We implement the Actor Model using Guile Fibers, where each actor:

1. Has a unique identity
2. Maintains private state
3. Communicates only through message passing
4. Can spawn new actors
5. Processes messages sequentially

* Concurrency Model

** Fiber-based Concurrency

We use Guile Fibers for:

- *Lightweight threads*: Thousands of concurrent actors
- *CSP-style channels*: Type-safe communication
- *Cooperative scheduling*: Predictable performance
- *Non-blocking I/O*: Efficient resource utilization

** Write-Ahead Logging

All state changes are logged before application:

1. Write intent to WAL
2. Apply change to in-memory state
3. Asynchronously flush to persistent storage
4. Compact WAL periodically

* Performance Considerations

** Optimization Strategies

1. *Message Batching*: Aggregate small messages
2. *Connection Pooling*: Reuse network connections
3. *Lazy Evaluation*: Defer computation until needed
4. *Memoization*: Cache frequently computed values
5. *Load Balancing*: Distribute work evenly

- Message throughput (msgs/sec)

* Security Model

** Authentication & Authorization

#+BEGIN_SRC mermaid :file auth-flow.png :exports results

    Auth-->>Client: auth_token
    
    Client->>Orchestrator: request(token, action)
    Orchestrator->>Auth: validate_token(token)
    Auth-->>Orchestrator: permissions
    
    Orchestrator->>Orchestrator: check_permission(action)
    Orchestrator->>Audit: log(client, action)

    end
